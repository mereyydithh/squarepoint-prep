PART 1: PYTHON OOP — FOUR PILLARS

1. Encapsulation (hide internal state)

class RiskLimit:
    def __init__(self, max_var: float):
        self.__max_var = max_var        # private via name-mangling
        self.__current_var = 0.0

    def is_breached(self) -> bool:
        return self.__current_var > self.__max_var

2. Inheritance (parent → child)

class Instrument:
    def market_value(self):
        raise NotImplementedError

class Equity(Instrument):
    def market_value(self):
        return self.shares * self.price

class Bond(Instrument):
    def market_value(self):
        return self.face * (self.clean_price + self.accrued) / 100

3. Polymorphism (same interface, different behavior)

portfolio = [Equity(...), Bond(...)]
total_mv = sum(inst.market_value() for inst in portfolio)  # works for any Instrument

4. Abstraction (hide complexity)

from abc import ABC, abstractmethod

class RiskModel(ABC):
    @abstractmethod
    def compute_var(self, positions, confidence: float) -> float:
        pass

class HistoricalVaR(RiskModel):
    def compute_var(self, positions, confidence=0.99):
        returns = self._get_returns(positions)
        return -np.percentile(returns, (1 - confidence) * 100)

Concept Use case

PART 4: RISK-SPECIFIC PYTHON CODE

Historical VaR

def historical_var(returns: pd.Series, confidence: float = 0.99,
                   portfolio_value: float = 1_000_000) -> float:
    sorted_returns = returns.sort_values()
    index = int(len(sorted_returns) * (1 - confidence))
    var_return = sorted_returns.iloc[index]
    return -var_return * portfolio_value

Sharpe Ratio

def sharpe_ratio(returns: pd.Series, risk_free_rate: float = 0.0375) -> float:
    excess_returns = returns - risk_free_rate / 252
    return excess_returns.mean() / excess_returns.std() * np.sqrt(252)

Find Drawdown Periods

def find_drawdowns(cumulative_pnl: pd.Series, top_n: int = 3):
    peak = cumulative_pnl.cummax()
    drawdown = cumulative_pnl - peak
    is_drawdown = drawdown < 0
    groups = (~is_drawdown).cumsum()
    worst = drawdown[is_drawdown].groupby(groups).min().nsmallest(top_n)
    return worst

        start = time.perf_counter()

PART 5: SQL FUNDAMENTALS

Core Queries

-- GROUP BY + HAVING: net position per ticker
SELECT ticker, SUM(quantity) as net_qty
FROM positions
WHERE book = 'DISC_MACRO'
GROUP BY ticker
HAVING SUM(quantity) <> 0
ORDER BY net_qty DESC;

-- JOIN: positions with market data
SELECT p.ticker, p.quantity, m.last_price,
       p.quantity * m.last_price as market_value
FROM positions p
INNER JOIN market_data m ON p.ticker = m.ticker;

-- Window functions: running P&L + rank
SELECT ticker, pnl,
       SUM(pnl) OVER (PARTITION BY ticker ORDER BY trade_date) as cumulative_pnl,
       RANK() OVER (ORDER BY pnl DESC) as pnl_rank
FROM daily_pnl;

-- CTE: annualized vol per ticker
WITH daily_returns AS (
    SELECT ticker,
           (close - LAG(close) OVER (PARTITION BY ticker ORDER BY date))
           / LAG(close) OVER (PARTITION BY ticker ORDER BY date) as ret
    FROM prices
)
SELECT ticker, STDDEV(ret) * SQRT(252) as annual_vol
FROM daily_returns
GROUP BY ticker;

Key Concepts

Concept Explanation
INNER JOIN Only matching rows from both tables
LEFT JOIN All rows from left + matches from right (NULLs for non-matches)
FULL OUTER JOIN All rows from both tables
GROUP BY + HAVING GROUP BY aggregates; HAVING filters AFTER aggregation (vs WHERE before)
Window functions OVER (PARTITION BY ... ORDER BY ...) — aggregate without collapsing rows
Key window funcs ROW_NUMBER, RANK, LAG, LEAD, SUM, AVG
Indexes B-tree on ticker → O(log n) lookup vs O(n) scan
ACID Atomicity, Consistency, Isolation, Durability

PART 6: kdb+/q (AWARENESS — NOT DEEP)

Why kdb+ Exists

  • Column-oriented: aggregations over billions of rows in milliseconds.
  • In-memory: data lives in RAM.
  • Time-series native: as-of joins, sliding windows, fills are first-class operations.

Core Syntax

// Read CSV
trades: ("DSFJI"; enlist ",") 0: `:trades.csv

// SQL-like query
select avg price, sum size by sym from trades where date = 2026.04.07

// VWAP (volume-weighted average price)
select vwap: size wavg price by sym from trades

// As-of join: last quote at each trade time
aj[`sym`time; trades; quotes]

// Rolling 20-day volatility
update vol20: 20 mdev log ratios close by sym from daily

What to Say If Asked

"I used kdb+ at BNP NYC for tick-level market data — FX forwards, swaps, corp hedging. The key advantage is speed: aggregations over millions of rows interactively that would hang in SQL. As-of joins are essential for aligning trade timestamps to market snapshots for P&L attribution."

PART 8: INTERVIEW APPLICATION

If asked "How would you build a daily risk report?":

"I'd use Python with pandas for data manipulation and numpy for calculations. The pipeline:

  1. Data ingestion: Pull positions from the risk database (SQL query with JOINs to get positions + market data). Pull market data from Bloomberg API or internal feed.

  2. Risk calculations: Compute portfolio-level metrics — VaR (historical: sort returns, take percentile), Greeks (delta, gamma, vega per position, summed for portfolio), exposure by sector/factor.

  3. Stress tests: Apply shock scenarios to the portfolio — rate moves, equity drops, vol spikes. Pre-defined scenarios plus ad-hoc ones for current events.

  4. Output: Format into a report — summary dashboard (total VaR, limit utilization, P&L attribution), detailed position-level breakdown, and any limit breaches highlighted.

At BNP I built exactly this — a live stress-testing model that ran 100+ scenarios daily and provided hedging signals to traders. Python OOP design: each risk model as a class inheriting from a base RiskModel interface, making it easy to add new stress scenarios without modifying existing code."
